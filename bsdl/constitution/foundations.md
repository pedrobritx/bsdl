# Foundations

## 1. Worldview

Reality exists independently of us. Software does not. Software is invented, like literature, and becomes meaningful through human interpretation, use, and maintenance.

That means software is not discovered the way a mineral is discovered. It is created the way a book, a law, a city plan, or a language pattern is created.

### Implication

Software can be judged only in context. It is not an object floating in pure abstraction. It lives inside history, institutions, users, budgets, and dependencies.

## 2. Ontology

Software is located in human intention, language, and executable material form.

It appears in different layers:

- as an idea in a human mind
- as language in a conversation, spec, or design note
- as code in a repository
- as a binary or runtime behavior
- as an effect in the world

It may persist after humanity disappears, but then it loses its social purpose. It remains physically present while becoming functionally meaningless.

### Conceptual note

Software is not only code. Code is one manifestation of software, not the whole of it. The same is true of documentation, tests, binaries, deployments, and observed behavior.

## 3. Epistemology

Truth in software is determined by the convergence of intent, specification, implementation, empirical validation, and observed behaviour. If any diverge, truth is incomplete.

Software truth therefore has layers:

- **Intent**: what the system is trying to do
- **Specification**: what the system is declared to do
- **Implementation**: what the system is built to do
- **Validation**: what tests prove it can do
- **Observed behaviour**: what it actually does in reality

When these layers disagree, the system is epistemically broken. Not necessarily unusable, but no longer trustworthy.

### Why tests outrank reasoning

Reasoning is necessary, but it is incomplete. Tests are more trustworthy than reasoning because reasoning is limited by memory, bias, incomplete context, and hidden assumptions.

Reasoning proposes; evidence disposes.

### Verification hierarchy

1. **Automated tests** check expected behavior.
2. **Static analysis and types** catch structural violations before runtime.
3. **Formal methods** prove properties in high-risk domains.
4. **Observability** reveals production reality.

## 4. Axiology

The primary values of software, in descending order, are:

1. Knowledge
2. Justice
3. Equality
4. Freedom
5. Happiness
6. Autonomy

These are not decorative values. They are priorities that constrain later design.

### Interpretation

- **Knowledge** is primary because software is a knowledge system before it is anything else.
- **Justice** matters because software can centralize or redistribute power.
- **Equality** matters because access, opportunity, and legibility are never neutral.
- **Freedom** matters because software can liberate or capture behavior.
- **Happiness** matters because systems should improve lived experience, not merely efficiency.
- **Autonomy** matters because users should retain meaningful agency.

## 5. Ethics

Accessibility is not optional. Software should protect users from themselves when they are likely to be wrong or harmed by their own choices. Software should refuse unethical requests.

This means software is not morally neutral. It may assist, but it should not collaborate in harm.

### Ethical boundary examples

- A banking system should not enable predatory exploitation.
- A learning tool should not exclude users with disabilities.
- A platform should not intentionally manipulate users into compulsive behavior.
- A system should not hide or amplify harm merely because it is profitable.

## 6. Society and labor

Software is collectively created. No software is built by one person alone. Individuals create it, but always inside a society that supplies language, tools, infrastructure, and prior knowledge.

In the broad sense, software belongs to society after its immediate creators. Ownership may be locally constrained by law or contract, but the philosophical origin of software is collective.

### Consequence

A private implementation can be ethically tolerated as a historical compromise, but private ownership is never a final philosophical justification by itself.
